# Requirements Document

## Introduction

The DSA Mock Interviewer is a Kiro-native feature that turns Kiro's built-in AI model into a FAANG-level technical interviewer. It runs entirely inside the IDE — no browser, no React app, no external API keys. Kiro acts as the interviewer using a steering file that defines its persona and rules. A user-triggered hook starts a session. Problems are either provided by the Candidate directly or generated by the Interviewer based on the selected company, topic, and difficulty filters. The feature replicates the full mock interview experience: structured phases, Socratic coaching, hint tracking, JS pitfall detection, and a scored debrief — all through Kiro's chat interface.

---

## Glossary

- **Interviewer**: Kiro's AI model operating under the interviewer persona defined in the steering file.
- **Candidate**: The user participating in the mock interview session.
- **Session**: A single end-to-end mock interview from problem presentation through debrief.
- **Steering_File**: The `.kiro/steering/dsa-interviewer.md` file that defines the Interviewer's persona, rules, and phase logic — always active in Kiro chat.
- **Hook**: The `.kiro/hooks/start-mock-interview.md` user-triggered hook that initialises a session.
- **Practice_Mode**: A session mode where the Interviewer provides hints freely and is coaching-oriented.
- **Mock_Mode**: A session mode that simulates a real FAANG interview — strict pacing, no unsolicited hints, pushback on suboptimal solutions.
- **Phase**: A distinct stage within a session (Clarification, Approach, Coding, Edge_Case_Challenge, Dry_Run, Debrief).
- **Dry_Run**: The phase between Coding and Debrief where the Candidate traces through their solution step by step with a specific input.
- **Hint**: A Socratic nudge provided by the Interviewer that does not reveal the solution directly.
- **Pushback**: An Interviewer challenge such as "can you optimise that?" or "what's the time complexity?".
- **Debrief**: The scored post-coding review covering seven dimensions.
- **Think_Aloud_Score**: A scoring dimension that evaluates the Candidate's ability to verbalise tradeoffs, consider multiple approaches, and articulate reasoning during the session.
- **Follow_Up_Problem**: A harder variant of the session problem presented after the Debrief to train adaptability.
- **DSA_Pattern**: A recognised algorithmic pattern (e.g., sliding window, monotonic stack, two pointers) that a problem maps to.
- **Communication_Anti_Pattern**: A detectable negative communication behaviour during an interview (e.g., prolonged silence, jumping to code without explanation, hedging excessively).
- **Candidate_Problem_Mode**: A session mode where the Candidate provides the problem directly instead of having the Interviewer generate one.
- **Multi_Round_Session**: A session format simulating a full interview loop with multiple rounds of increasing difficulty.
- **Weakness_Log**: A persistent markdown file that tracks the Candidate's weak areas across multiple sessions.
- **Optimal_Solution_Walkthrough**: An on-demand post-debrief explanation of the optimal solution, triggered by the `!optimal` command.
- **Pace_Report**: A timing breakdown of how long the Candidate spent in each phase, compared against ideal FAANG pacing benchmarks.
- **Difficulty_Progression**: An adaptive system that adjusts problem difficulty across sessions based on the Candidate's past performance.
- **Edge_Case_Challenge**: A mini-phase after coding where the Interviewer presents tricky edge cases one by one and the Candidate must determine whether their code handles them.
- **Session_Replay**: A concise markdown summary of an entire session saved to a file for later review.
- **Interviewer_Personality**: A configurable interviewer style that affects tone and behaviour (e.g., friendly, neutral, tough).
- **JS_Pitfall**: A JavaScript-specific coding mistake the Interviewer is trained to detect (e.g., `=` vs `===`, missing `let`/`const`, off-by-one errors).
- **Complexity_Analysis**: The Candidate's verbal or written explanation of time and space complexity.
- **SDE-2**: Software Development Engineer level 2 with 3 to 3.5 years of experience — the target difficulty calibration for problems and evaluation, specifically for MAANG/FAANG-level interviews.

## Requirements

### Requirement 1: Interviewer Persona via Steering File

**User Story:** As a Candidate, I want Kiro to behave as a consistent FAANG-level technical interviewer throughout my session, so that the experience feels realistic and structured.

#### Acceptance Criteria

1. THE Steering_File SHALL define the Interviewer persona as a MAANG/FAANG SDE-2 (3-3.5 YOE) technical interviewer with a Socratic, non-solution-giving style.
2. THE Steering_File SHALL instruct the Interviewer to never reveal a complete solution unless the Candidate explicitly types a designated reveal command (e.g., `!reveal`).
3. THE Steering_File SHALL define the six session phases in order: Clarification → Approach → Coding → Edge_Case_Challenge → Dry_Run → Debrief.
4. THE Steering_File SHALL specify that the Interviewer must ask at least two clarifying questions before allowing the Candidate to proceed to the Approach phase.
5. WHILE in Mock_Mode, THE Steering_File SHALL instruct the Interviewer to withhold hints unless the Candidate explicitly requests one via a designated hint command (e.g., `!hint`).
6. WHILE in Practice_Mode, THE Steering_File SHALL instruct the Interviewer to proactively offer guidance when the Candidate appears stuck for more than two exchanges.
7. THE Steering_File SHALL instruct the Interviewer to maintain the interviewer persona consistently across all chat turns within a session.

---

### Requirement 2: Session Initialisation via Hook

**User Story:** As a Candidate, I want to start a mock interview session with a single command, so that I can begin practising without manual setup.

#### Acceptance Criteria

1. THE Hook SHALL be a user-triggered hook located at `.kiro/hooks/start-mock-interview.md`.
2. WHEN the Candidate triggers the Hook, THE Interviewer SHALL prompt the Candidate to select a session mode (Practice_Mode or Mock_Mode).
3. WHEN the Candidate triggers the Hook, THE Interviewer SHALL prompt the Candidate to optionally select a target company (Google, Meta, Amazon, Microsoft, Apple, or Random).
4. WHEN the Candidate triggers the Hook, THE Interviewer SHALL prompt the Candidate to optionally select a topic (Arrays, DP, Graphs, Trees, Strings, Linked Lists, Backtracking, Heap, or Random).
5. WHEN Mock_Mode is selected, THE Interviewer SHALL prompt the Candidate to select a timer duration (20, 35, or 45 minutes) and SHALL display the selected duration at session start.
6. AFTER the Candidate completes all selections, THE Interviewer SHALL generate a problem matching the chosen company, topic, and difficulty filters (calibrated to SDE-2 level, 3-3.5 YOE, MAANG/FAANG bar) and SHALL present the full problem statement with examples and constraints to the Candidate.
7. IF the Candidate provides no company or topic preference, THEN THE Interviewer SHALL generate a random SDE-2 (3-3.5 YOE) calibrated problem from any major DSA topic.
8. EACH generated problem SHALL include: a problem title, difficulty (Easy/Medium/Hard), a clear problem statement, at least two example inputs and outputs, and constraints.

---

### Requirement 3: Structured Interview Phases

**User Story:** As a Candidate, I want the interview to follow a realistic phase structure, so that I practise the full interview workflow, not just coding.

#### Acceptance Criteria

1. THE Interviewer SHALL conduct every session in exactly six sequential phases: Clarification, Approach, Coding, Edge_Case_Challenge, Dry_Run, and Debrief.
2. WHEN the Clarification phase begins, THE Interviewer SHALL ask the Candidate at least two clarifying questions about the problem (e.g., input constraints, edge cases, expected output format).
3. WHEN the Candidate has answered the clarifying questions, THE Interviewer SHALL transition to the Approach phase by explicitly stating the phase change.
4. WHILE in the Approach phase, THE Interviewer SHALL ask the Candidate to describe their intended algorithm before writing any code.
5. WHEN the Candidate describes a suboptimal approach, THE Interviewer SHALL issue a Pushback asking whether a more efficient solution exists before allowing the Candidate to proceed to Coding.
6. WHEN the Candidate begins the Coding phase, THE Interviewer SHALL monitor the Candidate's code for JS_Pitfalls and SHALL flag each one as it is detected.
7. WHEN the Coding phase ends (Candidate signals completion or timer expires in Mock_Mode), THE Interviewer SHALL transition to the Edge_Case_Challenge phase by explicitly stating the phase change.
8. WHEN the Edge_Case_Challenge phase ends, THE Interviewer SHALL transition to the Dry_Run phase by explicitly stating the phase change.
9. WHEN the Dry_Run phase begins, THE Interviewer SHALL provide a specific test input and ask the Candidate to trace through their solution step by step, stating the value of key variables at each iteration.
10. WHILE in the Dry_Run phase, THE Interviewer SHALL compare the Candidate's stated trace against the expected behaviour of the Candidate's code and SHALL flag any discrepancy between the trace and the actual code logic.
11. WHEN the Dry_Run phase ends, THE Interviewer SHALL transition to the Debrief phase.

---

### Requirement 4: Hint System

**User Story:** As a Candidate, I want a controlled hint system, so that I can get unstuck without having the solution handed to me.

#### Acceptance Criteria

1. THE Interviewer SHALL track the total number of hints requested during a session and SHALL display the hint count in the Debrief.
2. WHEN the Candidate types `!hint`, THE Interviewer SHALL provide one Socratic nudge that guides the Candidate toward the next logical step without revealing the solution.
3. THE Interviewer SHALL NOT provide more than three hints per session in Mock_Mode; IF the Candidate requests a fourth hint in Mock_Mode, THEN THE Interviewer SHALL decline and note that the hint limit has been reached.
4. WHILE in Practice_Mode, THE Interviewer SHALL provide hints without a count limit.
5. EACH hint SHALL be distinct from previous hints given in the same session and SHALL progress the Candidate closer to the solution with each successive hint.

---

### Requirement 5: Interviewer Pushbacks

**User Story:** As a Candidate, I want the Interviewer to challenge my solutions and thinking, so that I practise handling real interview pressure.

#### Acceptance Criteria

1. WHEN the Candidate proposes a solution with non-optimal time complexity, THE Interviewer SHALL ask "Can you optimise this further?" and SHALL specify the current complexity the Candidate stated.
2. WHEN the Candidate completes a coding block without stating complexity, THE Interviewer SHALL ask the Candidate to analyse the time and space complexity of their solution.
3. WHEN the Candidate's code does not handle at least one obvious edge case (empty input, single element, negative numbers where applicable), THE Interviewer SHALL prompt the Candidate to consider that edge case by name.
4. WHEN the Candidate provides a Complexity_Analysis, THE Interviewer SHALL confirm whether it is correct or SHALL provide a correction with a brief explanation.
5. THE Interviewer SHALL issue at least one Pushback per session during the Coding or Approach phase.

---

### Requirement 6: JavaScript Pitfall Detection

**User Story:** As a Candidate writing JavaScript, I want the Interviewer to catch common JS mistakes in my code, so that I avoid them in real interviews.

#### Acceptance Criteria

1. THE Steering_File SHALL list the JS_Pitfalls the Interviewer is trained to detect: use of `=` instead of `===` in conditions, undeclared variables (missing `let`, `const`, or `var`), off-by-one errors in loop bounds, mutating function parameters directly, and using `==` instead of `===`.
2. WHEN the Candidate writes code containing a JS_Pitfall, THE Interviewer SHALL flag it immediately by quoting the offending line and naming the pitfall category.
3. THE Interviewer SHALL NOT fix the JS_Pitfall for the Candidate; THE Interviewer SHALL ask the Candidate to identify and correct it.
4. WHEN the Candidate corrects a flagged JS_Pitfall, THE Interviewer SHALL acknowledge the correction and continue the session.

---

### Requirement 7: Mock Mode Timer

**User Story:** As a Candidate in Mock_Mode, I want a time-boxed session, so that I practise working under real interview time pressure.

#### Acceptance Criteria

1. WHEN Mock_Mode is selected, THE Interviewer SHALL record the selected timer duration (20, 35, or 45 minutes) and SHALL state the start time and expected end time in the first message of the session.
2. THE Interviewer SHALL issue a time-check reminder when approximately half the session time has elapsed (e.g., "You have 17 minutes remaining").
3. THE Interviewer SHALL issue a final warning when 5 minutes remain in the session.
4. WHEN the timer expires, THE Interviewer SHALL announce that time is up and SHALL immediately transition to the Debrief phase regardless of whether the Candidate has finished coding.
5. IF the Candidate has not reached the Coding phase when the timer expires, THEN THE Interviewer SHALL note this in the Debrief as a time management observation.

---

### Requirement 8: Debrief and Scoring

**User Story:** As a Candidate, I want a structured debrief with scores after each session, so that I know exactly what to improve.

#### Acceptance Criteria

1. THE Interviewer SHALL produce a Debrief at the end of every session covering exactly seven scoring dimensions: Approach Quality, Time/Space Complexity Accuracy, Edge Case Coverage, Communication & Fluency, Clarifying Questions Quality, Think_Aloud_Score, and Code Narration Quality.
2. EACH scoring dimension SHALL be rated on a scale of 1 to 5, where 1 is poor and 5 is excellent, and SHALL include a one-sentence justification.
3. THE Think_Aloud_Score SHALL evaluate whether the Candidate verbalised tradeoffs between approaches, considered multiple strategies before committing, and articulated reasoning at each decision point during the session.
4. THE Debrief SHALL include the total hint count used during the session.
5. THE Debrief SHALL include an overall session verdict: "Hire", "Borderline", or "No Hire", based on the aggregate score across all seven dimensions.
6. THE Debrief SHALL include at least one specific, actionable improvement suggestion for the Candidate.
7. WHERE the Candidate's solution contained JS_Pitfalls, THE Debrief SHALL list each pitfall detected and whether it was corrected during the session.
8. WHEN the session was in Mock_Mode, THE Debrief SHALL include a time management note indicating whether the Candidate completed the solution within the allotted time.

---

### Requirement 9: Company-Specific Problem Selection

**User Story:** As a Candidate targeting a specific FAANG company, I want the Interviewer to select problems and apply evaluation criteria relevant to that company, so that my practice is targeted.

#### Acceptance Criteria

1. WHEN the Candidate selects Google as the target company, THE Interviewer SHALL prefer problems tagged with DP or Graphs and SHALL apply Google-style evaluation emphasis on optimal complexity and multiple approaches.
2. WHEN the Candidate selects Meta as the target company, THE Interviewer SHALL prefer problems tagged with Trees or Graphs and SHALL apply Meta-style evaluation emphasis on speed and production-quality code.
3. WHEN the Candidate selects Amazon as the target company, THE Interviewer SHALL prefer problems tagged with DP, Trees, or Design and SHALL apply Amazon-style evaluation emphasis on edge case handling and code correctness.
4. WHEN the Candidate selects Microsoft as the target company, THE Interviewer SHALL prefer problems tagged with Trees or Backtracking and SHALL apply Microsoft-style evaluation emphasis on recursive thinking and iterative alternatives.
5. WHEN the Candidate selects Apple as the target company, THE Interviewer SHALL prefer problems tagged with Arrays or Design and SHALL apply Apple-style evaluation emphasis on code quality and attention to detail.
6. THE Debrief SHALL reference the selected company's known evaluation style when providing the overall verdict and improvement suggestions.

---

### Requirement 10: Session State Continuity

**User Story:** As a Candidate, I want the Interviewer to maintain consistent session context throughout the chat, so that the interview does not lose track of what has been discussed.

#### Acceptance Criteria

1. THE Interviewer SHALL track and reference the current phase explicitly at each phase transition.
2. THE Interviewer SHALL reference the Candidate's previously stated approach when issuing Pushbacks during the Coding phase.
3. THE Interviewer SHALL reference the specific problem title and constraints when asking clarifying questions or flagging edge cases.
4. IF the Candidate asks to restart the session, THEN THE Interviewer SHALL confirm the restart, reset the hint count to zero, and re-trigger the session initialisation flow from Requirement 2.
5. THE Interviewer SHALL maintain a running count of hints used and JS_Pitfalls flagged throughout the session and SHALL reference these counts accurately in the Debrief.


---

### Requirement 11: Follow-Up Problem Escalation

**User Story:** As a Candidate, I want the Interviewer to present a harder variant of the problem after I solve it, so that I practise adaptability under pressure like in real FAANG rounds.

#### Acceptance Criteria

1. WHEN the Candidate successfully completes the Debrief with a verdict of "Hire" or "Borderline", THE Interviewer SHALL offer a Follow_Up_Problem that is a harder variant of the original session problem.
2. THE Follow_Up_Problem SHALL add at least one additional constraint to the original problem (e.g., reduced space complexity requirement, sorted input assumption, or streaming input).
3. WHEN the Candidate accepts the Follow_Up_Problem, THE Interviewer SHALL run a condensed session covering only the Approach and Coding phases for the Follow_Up_Problem.
4. THE Interviewer SHALL explicitly state how the Follow_Up_Problem relates to the original problem and what new constraint has been added.
5. IF the Candidate declines the Follow_Up_Problem, THEN THE Interviewer SHALL acknowledge the decision and end the session.

---

### Requirement 12: Think Aloud Scoring

**User Story:** As a Candidate, I want explicit feedback on my thought process during the interview, so that I improve the verbal reasoning skills that are the primary reason candidates fail despite solving the problem.

#### Acceptance Criteria

1. THE Interviewer SHALL evaluate the Candidate's Think_Aloud_Score as a separate scoring dimension in the Debrief, rated 1 to 5 with a one-sentence justification.
2. THE Think_Aloud_Score SHALL assess three sub-criteria: whether the Candidate verbalised tradeoffs between approaches, whether the Candidate considered multiple strategies before committing to one, and whether the Candidate articulated reasoning at each decision point.
3. WHILE in the Approach phase, THE Interviewer SHALL prompt the Candidate to verbalise tradeoffs if the Candidate proposes an approach without discussing alternatives.
4. WHILE in the Coding phase, THE Interviewer SHALL note instances where the Candidate codes silently for more than three consecutive exchanges without explaining their reasoning.
5. THE Debrief SHALL include specific examples from the session where the Candidate demonstrated strong or weak think-aloud behaviour.

---

### Requirement 13: Dry Run Phase

**User Story:** As a Candidate, I want the Interviewer to ask me to trace through my solution with a specific input after coding, so that I practise the dry-run step standard at Google and Meta interviews.

#### Acceptance Criteria

1. THE Interviewer SHALL conduct the Dry_Run phase after the Coding phase and before the Debrief phase in every session.
2. WHEN the Dry_Run phase begins, THE Interviewer SHALL provide a specific test input (distinct from the examples in the problem statement) and ask the Candidate to trace through their code step by step.
3. THE Interviewer SHALL ask the Candidate to state the value of key variables at each significant step of the trace.
4. WHILE the Candidate is tracing, THE Interviewer SHALL compare the Candidate's stated variable values against the expected behaviour of the Candidate's written code and SHALL flag any discrepancy.
5. IF the Candidate's trace reveals a bug in their code, THEN THE Interviewer SHALL point out the discrepancy and give the Candidate an opportunity to fix the code before proceeding to Debrief.
6. WHEN the timer expires in Mock_Mode during the Dry_Run phase, THE Interviewer SHALL skip the remainder of the Dry_Run and transition directly to the Debrief phase.
7. THE Debrief SHALL include a Dry_Run accuracy note indicating whether the Candidate's trace matched their code's actual behaviour.

---

### Requirement 14: Pattern Recognition Training

**User Story:** As a Candidate, I want to learn which DSA pattern each problem maps to after the debrief, so that I build the "I've seen this shape before" instinct for real interviews.

#### Acceptance Criteria

1. AFTER the Debrief, THE Interviewer SHALL identify and state the primary DSA_Pattern the session problem maps to (e.g., sliding window, monotonic stack, two pointers, BFS/DFS, dynamic programming subproblem structure).
2. THE Interviewer SHALL provide a one-sentence explanation of why the problem maps to the identified DSA_Pattern.
3. THE Interviewer SHALL suggest two to three similar problems that share the same DSA_Pattern, providing the problem title and difficulty level for each.
4. EACH suggested similar problem SHALL include the problem title and difficulty level.
5. IF the session problem maps to multiple DSA_Patterns, THEN THE Interviewer SHALL list all applicable patterns and indicate which is the primary pattern.

---

### Requirement 15: Communication Anti-Pattern Detection

**User Story:** As a Candidate, I want the Interviewer to flag negative communication habits during the session, so that I eliminate real interview killers before they cost me an offer.

#### Acceptance Criteria

1. THE Interviewer SHALL detect the following Communication_Anti_Patterns during a session: prolonged silence (no substantive response for more than two consecutive exchanges), jumping to code without explaining the approach, excessive hedging (using phrases like "I think" or "maybe" more than three times without committing to a direction), and over-explaining obvious steps.
2. WHEN the Interviewer detects a Communication_Anti_Pattern, THE Interviewer SHALL flag it in real time with a brief, constructive note naming the anti-pattern and suggesting a correction.
3. THE Interviewer SHALL track the count of each Communication_Anti_Pattern type detected during the session.
4. THE Debrief SHALL include a Communication Anti-Patterns section listing each detected anti-pattern, the number of occurrences, and one actionable suggestion per anti-pattern type.
5. WHILE in Practice_Mode, THE Interviewer SHALL flag Communication_Anti_Patterns immediately when detected; WHILE in Mock_Mode, THE Interviewer SHALL note them silently and report them only in the Debrief.

---

### Requirement 16: Candidate-Provided Problem Mode

**User Story:** As a Candidate, I want to paste my own problem and have the Interviewer run a full session with it, so that I can practise with specific problems from my study list.

#### Acceptance Criteria

1. WHEN the Candidate triggers the Hook, THE Interviewer SHALL offer Candidate_Problem_Mode as an additional option alongside topic and company selection.
2. WHEN the Candidate selects Candidate_Problem_Mode, THE Interviewer SHALL prompt the Candidate to paste or describe the problem statement.
3. AFTER the Candidate provides the problem statement, THE Interviewer SHALL validate that the problem contains sufficient information to run a session (problem description, at least one example input/output) and SHALL request clarification if the problem statement is incomplete.
4. THE Interviewer SHALL run the full session (all six phases) using the Candidate-provided problem with the same rules, pushbacks, and scoring as any other session.
5. THE Interviewer SHALL infer appropriate difficulty, topic tags, and constraints from the Candidate-provided problem to maintain consistent evaluation.

---

### Requirement 17: Multi-Round Session

**User Story:** As a Candidate, I want to simulate a full multi-round interview loop with escalating difficulty, so that I build stamina for real 4-5 hour FAANG interview days.

#### Acceptance Criteria

1. WHEN the Candidate triggers the Hook, THE Interviewer SHALL offer Multi_Round_Session as a session format option.
2. THE Multi_Round_Session SHALL consist of exactly three rounds: Round 1 (Easy to Medium difficulty, 20-minute timer), Round 2 (Medium to Hard difficulty, 35-minute timer), and Round 3 (Hard difficulty or Design problem, 45-minute timer).
3. EACH round in a Multi_Round_Session SHALL follow the full six-phase structure (Clarification, Approach, Coding, Edge_Case_Challenge, Dry_Run, Debrief) independently.
4. BETWEEN rounds, THE Interviewer SHALL provide a brief round summary with scores before proceeding to the next round.
5. AFTER all three rounds are complete, THE Interviewer SHALL produce a cumulative Debrief that aggregates scores across all rounds, identifies trends (e.g., "performance declined under fatigue"), and provides an overall multi-round verdict.
6. THE cumulative Debrief SHALL include a stamina assessment noting whether the Candidate's performance was consistent, improved, or degraded across rounds.

---

### Requirement 18: Weakness Tracker Across Sessions

**User Story:** As a Candidate, I want the Interviewer to maintain a running log of my weak areas across multiple sessions, so that my practice becomes targeted improvement instead of random repetition.

#### Acceptance Criteria

1. THE Interviewer SHALL maintain a Weakness_Log as a markdown file located at `src/weakness-log.md`.
2. AFTER each session Debrief, THE Interviewer SHALL append an entry to the Weakness_Log containing: the session date, problem title, scoring dimensions rated 3 or below, detected Communication_Anti_Patterns, and specific weak areas identified.
3. THE Weakness_Log SHALL group entries by weakness category (e.g., "Graph Traversal Edge Cases", "Space Complexity Analysis", "Think Aloud Under Pressure").
4. WHEN a new session begins, THE Interviewer SHALL read the Weakness_Log and SHALL reference recurring weak areas when selecting problems or providing targeted pushbacks.
5. IF the Weakness_Log contains three or more entries with the same weakness category, THEN THE Interviewer SHALL explicitly recommend that the Candidate focus on that area and SHALL prefer problems that exercise the identified weakness.

---

### Requirement 19: Optimal Solution Comparison

**User Story:** As a Candidate, I want to see the optimal solution after my debrief and understand where my thinking diverged, so that I learn not just the answer but where my reasoning went off track.

#### Acceptance Criteria

1. WHEN the Candidate types `!optimal` after the Debrief, THE Interviewer SHALL walk through the optimal solution for the session problem step by step.
2. THE Optimal_Solution_Walkthrough SHALL explain the algorithmic approach, the time and space complexity, and why the approach is optimal for the given constraints.
3. THE Optimal_Solution_Walkthrough SHALL highlight specific points where the Candidate's approach diverged from the optimal solution and SHALL explain why the optimal path is preferable at each divergence point.
4. THE Interviewer SHALL NOT provide the Optimal_Solution_Walkthrough before the Debrief is complete; IF the Candidate types `!optimal` before the Debrief, THEN THE Interviewer SHALL inform the Candidate that the walkthrough is available only after the Debrief.
5. WHERE the Candidate's solution was already optimal, THE Interviewer SHALL confirm optimality and suggest alternative approaches for educational value.

---

### Requirement 20: Interview Pace Coaching

**User Story:** As a Candidate, I want timing feedback on how long I spent in each interview phase, so that I can calibrate my pacing against ideal FAANG interview benchmarks.

#### Acceptance Criteria

1. THE Interviewer SHALL track the approximate time the Candidate spends in each phase (Clarification, Approach, Coding, Dry_Run) during every session.
2. THE Debrief SHALL include a Pace_Report comparing the Candidate's phase durations against ideal FAANG pacing benchmarks: Clarification (3-5 minutes), Approach (5-8 minutes), Coding (15-20 minutes), Dry_Run (3-5 minutes).
3. WHEN the Candidate spends more than 150% of the benchmark upper bound on any single phase, THE Pace_Report SHALL flag that phase as "Over Time" with a specific recommendation to reduce time spent.
4. WHEN the Candidate spends less than 50% of the benchmark lower bound on any single phase, THE Pace_Report SHALL flag that phase as "Rushed" with a recommendation to invest more time.
5. WHEN the session is in Mock_Mode, THE Interviewer SHALL issue a mid-session pace check if the Candidate is still in the Clarification or Approach phase after 40% of the total session time has elapsed.

---

### Requirement 21: Difficulty Progression Mode

**User Story:** As a Candidate, I want the Interviewer to dynamically adjust problem difficulty across sessions based on my performance, so that I am always practising at the right challenge level instead of picking difficulty manually.

#### Acceptance Criteria

1. WHEN the Candidate selects Difficulty_Progression during session initialisation, THE Interviewer SHALL start the first session with a Medium difficulty problem.
2. IF the Candidate's previous session verdict was "Hire" AND the Candidate used zero or one hints, THEN THE Interviewer SHALL escalate the next session's problem difficulty by one level (Medium → Hard).
3. IF the Candidate's previous session verdict was "No Hire" OR the Candidate used three or more hints, THEN THE Interviewer SHALL de-escalate the next session's problem difficulty by one level (Hard → Medium, Medium → Easy).
4. IF the Candidate's previous session verdict was "Borderline", THEN THE Interviewer SHALL maintain the same difficulty level for the next session.
5. THE Interviewer SHALL reference the Weakness_Log to determine the difficulty adjustment and SHALL state the chosen difficulty and the reason for the adjustment at the start of each session.
6. THE Interviewer SHALL cap difficulty at Hard and floor it at Easy — it SHALL NOT escalate beyond Hard or de-escalate below Easy.

---

### Requirement 22: Edge Case Challenge Round

**User Story:** As a Candidate, I want the Interviewer to throw tricky edge cases at me after I submit my code, so that I build the habit of reasoning about boundaries before being prompted.

#### Acceptance Criteria

1. THE Interviewer SHALL conduct the Edge_Case_Challenge after the Coding phase and before the Dry_Run phase, making the session phases: Clarification → Approach → Coding → Edge_Case_Challenge → Dry_Run → Debrief.
2. WHEN the Edge_Case_Challenge begins, THE Interviewer SHALL present three to four edge cases one at a time, each targeting a different boundary condition relevant to the problem (e.g., empty input, single element, maximum constraint value, duplicate values, negative numbers).
3. FOR each edge case, THE Interviewer SHALL ask the Candidate: "Does your code handle this case? What would it return?"
4. THE Candidate SHALL respond with their prediction for each edge case before the Interviewer confirms or denies correctness.
5. IF the Candidate's prediction is incorrect for an edge case, THE Interviewer SHALL explain why and give the Candidate an opportunity to fix their code before proceeding.
6. THE Debrief SHALL include an Edge Case Accuracy score indicating how many of the presented edge cases the Candidate correctly predicted out of the total presented.
7. WHILE in Mock_Mode, THE Interviewer SHALL limit the Edge_Case_Challenge to 3 minutes; IF time runs out, THE Interviewer SHALL skip remaining edge cases and proceed to Dry_Run.

---

### Requirement 23: Session Replay Summary

**User Story:** As a Candidate, I want a concise written summary of each session saved to a file, so that I can review my interview performance over time and build a personal interview journal.

#### Acceptance Criteria

1. AFTER the Debrief is complete, THE Interviewer SHALL generate a Session_Replay as a markdown file saved to `src/session-replays/`.
2. THE Session_Replay file SHALL be named with the format `{date}-{problem-title-kebab-case}.md` (e.g., `2026-04-04-two-sum.md`).
3. THE Session_Replay SHALL include: the session date, mode (Practice or Mock), target company (if selected), problem title and statement, the Candidate's stated approach, key decisions and pivots made during the session, mistakes and corrections, all hint requests and responses, JS_Pitfalls detected, the full Debrief scores, the overall verdict, and the Pace_Report.
4. THE Session_Replay SHALL be concise — no more than 150 lines — summarising the session flow rather than reproducing the full chat transcript.
5. THE Interviewer SHALL inform the Candidate of the saved file path after generating the Session_Replay.

---

### Requirement 24: Interviewer Personality Variants

**User Story:** As a Candidate, I want to choose the Interviewer's personality style before a session, so that I can practise handling different interviewer temperaments and build resilience for real interviews.

#### Acceptance Criteria

1. WHEN the Candidate triggers the Hook, THE Interviewer SHALL prompt the Candidate to select an Interviewer_Personality: "Friendly" (encouraging, warm, supportive tone), "Neutral" (professional, balanced, matter-of-fact tone), or "Tough" (challenging, direct, high-pressure tone).
2. IF the Candidate does not select an Interviewer_Personality, THEN THE Interviewer SHALL default to "Neutral".
3. WHILE in "Friendly" mode, THE Interviewer SHALL use encouraging language, acknowledge good steps explicitly, and frame pushbacks as gentle suggestions (e.g., "Nice start — have you considered whether there's a way to reduce the space usage?").
4. WHILE in "Neutral" mode, THE Interviewer SHALL maintain a professional, balanced tone with minimal emotional language and direct factual pushbacks (e.g., "Your current approach is O(n²). Can you do better?").
5. WHILE in "Tough" mode, THE Interviewer SHALL use direct, high-pressure language, issue pushbacks more frequently, give shorter acknowledgments, and apply stricter evaluation in the Debrief (e.g., "That's brute force. I'd expect a stronger candidate to see the optimal approach immediately. Think harder.").
6. THE Interviewer SHALL maintain the selected Interviewer_Personality consistently throughout the entire session including the Debrief.
7. THE Debrief SHALL note which Interviewer_Personality was active during the session.
